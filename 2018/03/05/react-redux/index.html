react-redux 初探

前言

我想很多人都遇到过应用状态管理的问题，redux想必大家都知道，都用过，在和react结合起来用的时候，我想大家也都知道react-redux这个插件，通过他我们可以很方便的将redux集合进我们的react项目中，能够很好的在组件状态更新后，通过组件的this.props就能得到更新后的状态值。但估计不少人不会去深究背后的原因，为什么通过react-redux我们就能监听到状态的变更从而更新组件呢，下面我们对react-redux的初探，看看具体是怎么发生的。

Provider

在聊Provider之前，我们先去看下React的一个特性，在我们平常开发的过程中，我们估计都遇到父组件给子组件传递属性的过程。如果子组件嵌套层次很深，那你可能需要一层一层的往下传，这个过程感觉很繁琐，有没有一种方法，可以直接从父组件到想要数据的子组件，而不用中间通过层层传递，答案是肯定的，React 提供了功能强大的 context api,来帮助开发实现这一点。

Note:
A new, safe version of context is under development for the upcoming 16.3 release.
在接下来16.3版本中将会加入新开发的，更安全的context。

Context

虽然这个特性很方便，但是官方还是建议我们不要去使用它，原因主要有几下几点：

  1. 它是实验性质的api，在未来的版本中有可能发生变动。
  2. 在大部分情况下，考虑使用更好的状态管理工具，如Redux,Mobx。
  3. 如果你才刚开始接触React,不要去使用这个特性，因为我们需要养成正确模式去实现具体的业务。

如果你在看了上面三点，你还要坚持去用这个特性，那么最好在小范围使用，并能够与其他业务代码隔离开来，避免api的变更带来升级困难，从而降低影响范围。
