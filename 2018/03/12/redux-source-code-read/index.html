前言

在使用redux的过程中，我们知道 action, store ,reducer 就是整个redux的全部，但就是这么简单的模式，让我们整个应用的状态管理变得更加简单，清晰。通过store，我们将action，reducer，以及一些中间件结合到了一起，那具体是怎么做到的呢？我们先把中间件放到一边，因为中间件的源码涉及到一些函数编程的思想。

中间件

中间件主要被用于分离那些不属于你应用的核心业务逻辑的可被组合起来使用的代码，提供了一个分类处理 action 的机会，在 middleware 中你可以检阅每一个流过的 action，挑选出特定类型的 action 进行相应操作，给你一次改变 action 的机会。比如redux-thunk中间件，可以判断action是否是函数，如果是函数进行区分处理，这样就可以处理返回函数的actionCreator了。

再看源码之前，先回顾下 Array.reduce 和 Array.reduceRight方法。

这两个方法都可以提供两个参数，一个方法fn,一个初始值。
方法fn会接收到的参数如下：

  • preValue : 上一次调用回调返回的值，或者是提供的初始值
  • curValue : 数组中当前被处理的数组项
  • index : 当前数组项在数组中的索引值
  • array : 调用 reduce / reduceRight方法的数组

来看下中间件的写法

所以chain是这些方法的数组，（next）=> (action) => {// do something}

我们现在假设只有一个中间件方法，chain中只有一个元素如下：

let func1 = function (next) {
    return function(action) {
        // todo something
    }
}

那么compose(…chain)返回func1,然后执行func1(store.dispatch), 这个时候返回的函数是 （action）=> {} 就是新的dispatch 。

如果是多个中间件，那么 dispatch = f1(f2(f3(store.dispatch))))，所以为了保证后面的中间件能够执行到，请务必执行next(action)。

reducer

createReducer源代码比较简单，主要是将分支reducer按照key，执行结果对应于全局大的State的key，进行组合，最后返回一个rootReducer。具体源代码就不在这展示了，有兴趣的自己去看。

总结

整个源码中间件部分比较难理解，这和express中间件类似，也和web开发服务端中的filter很像。所以大家可以对比着理解。
